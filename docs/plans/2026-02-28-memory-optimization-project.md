## 实施计划

### 阶段一：Bug 修复（基础设施）

**目标**：修复基础问题，为后续优化打好基础

**顺序**：

1. **B3: `apply_operation` 返回值**
   - 原因：B4（原子性）依赖此改动
   - 改动：返回 `bool` 表示成功/失败
   - 影响范围：`consolidate()` 调用处

2. **B1: 验证逻辑边界情况**
   - 改动：空内容时也要验证 section header 存在
   - 测试：添加边界 case 测试

3. **B2 + B6: 日志格式 + 类型安全**
   - 改动：统一用 `{}` 格式化，添加 `isinstance` 检查
   - 影响范围：小，可并行

4. **B5: 关键词触发补全**
   - 改动：添加 "papers"、"文章" 等关键词
   - 注意：保持 trigger 的精确性，避免误触发

**验收**：所有测试通过，无回归

---

### 阶段二：核心优化（健壮性）

**目标**：实现 P0/P1 优化项

**顺序**：

1. **B4 → 批量操作原子性**
   - 实现 `apply_operations_atomic()` 方法
   - 流程：备份 → 执行 → 失败回滚
   - 修改 `consolidate()` 使用原子方法

2. **记忆去重**
   - 先实现精确匹配去重
   - 语义去重作为可选功能（需要 embedding）

**验收**：
- 批量操作失败时，文件恢复到初始状态
- 相同内容不会重复追加

| 风险 | 影响 | 缓解措施 |

| 去重误判 | 低 | 先精确匹配，阈值设保守值 |
