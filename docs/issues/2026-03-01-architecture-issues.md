## 一、高优先级问题

### 1.1 裸 except 子句（200+ 处）

- **位置**: 整个代码库
- **严重程度**: 高
- **描述**: 大量 `except Exception:` 和 `except:` 不区分错误类型，导致真实错误被掩盖，难以调试
- **示例**:
  ```python
  except Exception:
      pass
  ```
- **建议**: 使用具体异常类型，至少区分可恢复/不可恢复错误

---

### 1.2 全局环境变量污染

- **位置**: `providers/litellm_provider.py:67-80`
- **严重程度**: 高
- **描述**: 直接修改 `os.environ`，在多 provider 场景下会互相覆盖 API Key 等配置
- **建议**: 使用 contextvars 或在每次调用时传递参数

---

### 1.3 工具执行无超时机制

- **位置**: `agent/tools/base.py:42-53`
- **严重程度**: 高
- **描述**: `execute` 方法没有超时控制，可能导致工具永久阻塞
- **建议**: 在 ToolRegistry.execute 中添加全局超时

---

### 1.4 语义相似度检查效率低

- **位置**: `agent/memory.py:197-223`
- **严重程度**: 高
- **描述**: `_is_semantically_similar` 每次都调用 embedding API，且对每个段落单独 embedding，效率低下
- **建议**: 使用批量 embedding 和缓存

---

### 1.5 Session 缓存无容量限制

- **位置**: `session/manager.py:91`
- **严重程度**: 高
- **描述**: `self._cache` 字典无限增长，可能导致内存泄漏
- **建议**: 使用 LRU 缓存（如 `functools.lru_cache`）或添加 TTL

---

### 1.6 内存合并失败未清理部分写入

- **位置**: `agent/memory.py:580-582`
- **严重程度**: 高
- **描述**: 如果 `apply_operations_atomic` 中途失败，可能只写入部分内容，导致数据不一致
- **建议**: 改进事务机制或添加回滚

---

### 1.7 数据库连接非线程安全

- **位置**: `rag/store.py:161`
- **严重程度**: 高
- **描述**: SQLite 连接在多线程/协程环境下使用可能出问题
- **建议**: 确保所有访问都在同一事件循环，或使用连接池

---

### 1.8 LiteLLM 异常被静默处理

- **位置**: `providers/litellm_provider.py:227-232`
- **严重程度**: 高
- **描述**: 所有异常都返回错误消息而非抛出，可能掩盖严重问题
- **建议**: 对可恢复和不可恢复错误区分处理

---

### 1.9 MCP 工具超时未传递给底层

- **位置**: `agent/tools/mcp.py`
- **严重程度**: 中（提升为高）
- **描述**: 配置中有 `tool_timeout` 但 MCP 调用可能不受其控制
- **建议**: 确保超时应用到 MCP 调用

---

### 1.10 Provider 匹配逻辑复杂度高

- **位置**: `config/schema.py:423-463`
- **严重程度**: 中（提升为高）
- **描述**: `_match_provider` 方法有多个 fallback 逻辑，难以理解和维护
- **建议**: 拆分为独立的 ProviderMatcher 类

## 二、中优先级问题

### 2.1 Tool Result 截断可能导致信息丢失

- **位置**: `agent/loop.py:585-606`
- **描述**: `_TOOL_RESULT_MAX_CHARS = 500` 硬编码截断
- **建议**: 改为可配置参数，或使用摘要策略

---

### 2.2 Session 锁竞争可能导致性能问题

- **位置**: `agent/loop.py:429-434`
- **描述**: `_session_locks_lock` 是全局锁，高并发时可能成为瓶颈
- **建议**: 使用分片锁或改进锁粒度

---

### 2.3 RAG 工具初始化时机问题

- **位置**: `agent/loop.py:343, 643`
- **描述**: `process_direct` 方法每次都调用 `_init_rag()`，可能导致重复初始化
- **建议**: 使用单例模式或懒加载

---

### 2.4 内存合并超时时间硬编码

- **位置**: `agent/loop.py:527`
- **描述**: `asyncio.timeout(60)` 硬编码在代码中
- **建议**: 提取为配置参数

---

### 2.5 任务清理回调中的潜在竞态条件

- **位置**: `agent/loop.py:441-451`
- **描述**: callback 在 task 完成时调用但没有错误处理
- **建议**: 添加更健壮的任务追踪机制

---

### 2.6 精确匹配去重效率低

- **位置**: `agent/memory.py:180, 322-325, 329-332`
- **描述**: 大文件上 `content in existing` 是 O(n) 操作
- **建议**: 使用 hash 或 set 进行去重

---

### 2.7 向量搜索禁用后状态不一致

- **位置**: `rag/store.py:196-198, 466-467`
- **描述**: `_vector_enabled` 和 `_vector_disabled_at` 的更新分散在多个地方
- **建议**: 封装向量状态管理方法

---

### 2.8 RAG 缓存键使用 hash() 可能碰撞

- **位置**: `rag/store.py:1074`
- **描述**: 使用 Python 的 hash()，结果与进程相关且可能碰撞
- **建议**: 使用 hashlib.sha256

---

### 2.9 API Key 传递方式不一致

- **位置**: `providers/litellm_provider.py:208-210`
- **描述**: 有些地方用 env var，有些用参数传递
- **建议**: 统一使用参数传递

---

### 2.10 CLI 命令重复创建 AgentLoop 实例

- **位置**: 多处命令函数
- **描述**: 每个命令都创建新的 AgentLoop，配置重复加载

### 2.13 异常被吞掉但未记录上下文
